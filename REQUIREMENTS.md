# REQUIREMENTS.md — 요구사항 정의서

작성일: 2026-07-04 · 인터뷰 기반 확정본

## 1. 문제 정의 및 배경

- **문제**: 지도 앱으로 음식점 후보를 추릴 때 리뷰를 하나하나 읽고 비교하는 과정이 번거롭다. 특히 한 음식점 안에서 어떤 메뉴가 실제 이용객에게 추천/비추천되는지 한눈에 보기 어렵다.
- **기존 해결의 한계**: ① 직접 검색해 리뷰를 훑는 방식은 시간이 든다 ② SNS 광고 기반 정보는 광고주 편향이 있어 실제 이용객 평가가 반영되지 않는다 ③ 개인 리뷰어 큐레이션 앱은 단일 공급자 관점이라 복수 사용자 리뷰를 종합하지 못한다.
- **심각도**: 크리티컬하지 않은 사소한 불편. 따라서 저비용·저유지보수 구조가 필수 전제다.

## 2. 타겟 사용자 및 유스케이스

- **사용자**: 개발자 본인 및 소수 지인 (개인용). Telegram은 chat_id 허용목록, 웹은 초대코드제.
- **동기**: 데이트 장소 선정, "뭐 먹지?" 결정 과정의 효율화.
- **가치 우선순위**: 편의성 > 속도 > 비용.

### UC-1. 약속 장소·메뉴 선정 (핵심)
네이버지도 앱에서 음식점을 보다가 공유 버튼 → Telegram 봇으로 전송. 봇이 리뷰 요약(총평·장점·단점)과 메뉴별 추천도를 응답한다. 여러 후보를 빠르게 비교해 결정한다.

### UC-2. 중복 조회 시 캐시 응답
이미 조회했던 음식점의 URL을 다시 보내면, 봇이 저장된 요약을 **마지막 갱신 시점과 함께** 즉시 응답하고 `/update` 명령을 안내한다. `/update` 시 최신 리뷰로 재수집·재요약한다.

### UC-3. 개인 리뷰 기록 — **MVP 제외**
`/review`로 개인 감상 기록, 조회 시 병행 표시. 추후 확장 후보.

## 3. 핵심 기능 목록 (MVP)

| ID | 기능 | 설명 |
|----|------|------|
| F1 | URL 입력 처리 | 네이버지도 공유 텍스트에서 naver.me URL 추출, place_id 해석 |
| F2 | 리뷰 수집 | 해당 음식점의 최근 방문자 리뷰 50개 스크래핑 |
| F3 | 리뷰 요약 | Claude API로 총평·장점·단점 요약 |
| F4 | 메뉴별 추천도 | 리뷰에서 메뉴 언급을 집계해 추천/비추천/호불호 + 언급 횟수 제시 |
| F5 | 캐시 | 결과를 DynamoDB에 저장(만료 없음), 재조회 시 갱신 시점과 함께 즉시 응답 |
| F6 | /update | 캐시를 무시하고 재수집·재요약 |
| F7 | 접근 제어 | chat_id 허용목록 + webhook secret token |
| F8 | 안내 | /start, /help, 잘못된 입력 시 사용법 안내 |

**MVP 제외**: 장소명 텍스트 검색, 개인 리뷰 기록(UC-3), 다국어, 웹 UI, 수익화.
(웹 UI는 MVP 이후 Phase 5에서 별도 격리 스택으로 추가됨 — 설계 근거는 `docs/web-design.md`)

## 4. 기술 스택 및 구현 방향

- **언어**: Python 3.12
- **인프라**: AWS SAM 서버리스 — Lambda 2개(WebhookFunction: 검증·즉답 / WorkerFunction: 수집·분석·발송), API Gateway HTTP API, DynamoDB(PAY_PER_REQUEST), Secrets Manager. 리전 ap-northeast-2.
- **프론트**: MVP는 Telegram 봇(webhook 방식)이 유일한 인터페이스. Phase 5에서 웹 진입점(Next.js 15 정적 PWA + Vercel, 별도 SAM 스택) 추가.
- **스크래핑**: 공식 API 부재 → httpx로 비공식 JSON/GraphQL 엔드포인트 직접 호출 우선. 막히면 Playwright(Lambda 컨테이너) 전환.
- **LLM**: Claude API (`anthropic` SDK), 호출 1회로 요약+메뉴 추천도 구조화 출력.
- **개발 체제**: 1인 개발. Advisor(설계·검증)/Worker(구현) 역할 분담 — CLAUDE.md 참조.

## 5. 리스크 및 고려사항

| 리스크 | 영향 | 대응 |
|--------|------|------|
| 네이버 비공식 엔드포인트 변경·차단 | 서비스 전면 중단 | 실측 덤프 기반 확정(Phase 1 게이트), 파서 분리로 교체 용이화, 개발자 에러 알림. 429·UA 실측 상세는 `experiments/findings.md` |
| Telegram webhook 3초 제한 vs 처리 10~20초 | 재시도 폭주 | 2-Lambda 분리, webhook 즉시 200 |
| Claude 비용 | 소액이지만 누적 | 캐시 우선 응답, 호출당 리뷰 50개 상한, 킬 스위치 상수. **실측 ~$0.04/신규 분석, 캐시 히트 무료** (PRD §9) |
| 스크래핑 요청 과다 | 차단 위험 | 개인용 소량 트래픽, RATE_LIMIT_DELAY, 캐시로 재수집 최소화 |
| 개인정보 | 낮음 | 수집 데이터는 공개 리뷰 텍스트뿐. 리뷰어 식별 정보는 저장하지 않음 |

## 6. 우선순위 및 개발 로드맵 (요약)

1. **Phase 0** — 문서·골격 재구축 (ref 이식, 2-Lambda 반영)
2. **Phase 1** — 스크래핑 실측 확정 + collector 구현 ← **게이트: 실패 시 이후 진행 불가**
3. **Phase 2** — Claude 분석(요약+메뉴 추천도)·응답 포맷
4. **Phase 3** — 캐시·/update 흐름
5. **Phase 4** — SAM 배포·실기기 E2E

상세 태스크와 진행 상태는 ROADMAP.md가 단일 진실 공급원이다.

## 7. 성공 기준

- URL 공유 → 요약 수신까지 신규 조회 30초 이내, 캐시 히트 3초 이내.
- 메뉴 추천도가 실제 리뷰 내용과 부합 (개발자 육안 검증).
- 이용자(본인·지인) 만족도. VOC 수집 후 추가 개발/유지보수 계획 수립.
