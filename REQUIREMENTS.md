# REQUIREMENTS.md — 사용자 원본 요구사항

> 문서 위계 최상위. 문제 정의·유스케이스·인터랙션 모델을 사용자 언어로 기술한다.
> (아래는 골격 스텁 — 실제 요구사항으로 채운다.)

## 1. 문제 정의

<!-- 왜 이 봇이 필요한가? 어떤 불편을 해결하는가? -->
- 예: 특정 장소(맛집·카페)의 리뷰를 일일이 스크롤하기 번거롭다. Telegram으로 이름만 보내면
  핵심 요약을 받고 싶다.

## 2. 대상 사용자

<!-- 누가 쓰는가? (개인용/허용된 소수) -->
- 예: 본인 및 허용 목록에 등록한 지인 몇 명(Chat ID 허용목록으로 제한).

## 3. 유스케이스

- UC-1: 사용자가 장소명을 보낸다 → 봇이 리뷰 요약 + 원문 일부를 응답한다.
- UC-2: 사용자가 네이버 지도 링크를 보낸다 → 동일하게 응답한다.
- UC-3: 사용자가 `/help`를 보낸다 → 사용법을 응답한다.

## 4. 인터랙션 모델

- 채널: Telegram (봇 1:1 대화). 프론트엔드 없음.
- 방식: 온디맨드 요청-응답 (스케줄 푸시 없음).

## 5. 비기능 요구 / 제약

- 개인 규모(저비용). 서버리스 온디맨드 과금.
- 허용 목록 외 사용자 요청은 무시. Webhook secret token 검증.
- 응답 지연은 수 초 이내 목표(무거우면 비동기 처리 고려).

## 6. 범위 밖 (Out of Scope)

- 예: 리뷰 작성/평점 등록, 다국어, 웹 UI 등.
