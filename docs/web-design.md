# web-design.md — 웹 진입점 설계 결정 기록

웹 진입점(`naver-review-web` 스택 + Vercel 프론트, 2026-07-07 라이브)의 **설계 결정과 그 근거**를 기록한다.
"무엇이 배포돼 있는가"는 `ARCHITECTURE.md`, "어떻게 배포하는가"는 `docs/setup-guide.md` §8이 원천이다. 이 문서는 "왜 이렇게 설계했는가"만 담는다.

## 배경과 절대 제약

Telegram 봇은 개인용으로 잘 동작하나, 주변에 Telegram 미사용자가 많아 공유 진입장벽이 있었다. 웹 진입점을 추가하되 두 가지 절대 제약을 둔다:

1. **기존 Telegram 봇 영향 0** — 다운타임·회귀·리소스 경합 금지.
2. **비용 통제** — Telegram은 chat_id 화이트리스트로 LLM 호출자가 한정되지만, URL 기반 웹은 링크가 퍼지면 토큰 비용이 폭증할 수 있다.

## 결정 1 — 격리 전략: 별도 SAM 스택 + 코어 소스 재사용

**채택**: 별도 CloudFormation 스택 `naver-review-web`(별도 Lambda·API GW·IAM·테이블). 코어 모듈(`naver_review_collector`·`review_analyst`·`config`)은 소스 레벨 공유.

**근거**:
- 같은 스택에 라우트를 추가하면 웹 재배포가 기존 함수를 건드려 제약 1 위반. 별도 스택이면 배포·런타임·IAM이 자연 격리된다.
- 완전 분리(코어 복제)는 격리가 최고지만 스크래핑 로직 이중 유지보수 비용이 크다. 코어 파이프라인이 이미 Telegram 비의존(`telegram_sender`·`review_formatter`만 Telegram 전용)이라 소스 재사용이 안전했다.
- **보안 공백 봉쇄**: Telegram `WorkerFunction`은 무인증이라 직접 invoke가 가능하면 전 방어를 우회한다. 웹 스택 IAM에 Telegram WorkerFunction invoke 권한을 **부여하지 않음**으로써 자연 차단.
- `config.py`는 부재 시크릿 키를 빈 문자열로 처리(`_secret.get(k, "")`)하므로, 웹 스택이 자체 시크릿(`naver-review/web`)만 담아도 크래시 없음 — 시크릿 격리의 핵심.

## 결정 2 — 응답 구조: 비동기 잡 + 폴링

**채택**: `POST /analyze`가 잡 생성 후 즉시 `job_id` 반환 → WebWorker가 비동기로 파이프라인 수행·잡에 결과 기록 → 프론트가 `GET /result/{job_id}` 폴링.

**근거**: 캐시 미스 파이프라인이 ~30초(수집+Claude 분석)로 **API Gateway 30초 타임아웃을 초과**한다. Telegram 스택의 2-Lambda 비동기 패턴(즉답 + Event invoke)을 웹에 맞게 계승한 형태. 잡은 `web_jobs` 테이블(TTL 1시간)에 저장.

## 결정 3 — 접근 제어·비용 방어 계층 (안쪽→바깥쪽)

| 계층 | 상태 | 내용 |
|---|---|---|
| ① 초대코드 게이트 | 운영 중 | `WEB_INVITE_CODES`(`{"코드":"표시이름"}`)로 유입 자체를 지인으로 한정. 코드 폐기·회전 가능, PII 불필요 |
| ② 서버측 세션 재검증 | 운영 중 | 매 API 호출마다 HMAC 세션 토큰 검증(`web_auth`) — 무인증 공백을 웹에서는 반드시 메움 |
| ③ read-through 공유 캐시 | 운영 중 | 웹 캐시 미스 시 Telegram `prod_review_cache`를 **읽기전용**으로 조회 → 워밍된 캐시 재사용으로 히트율↑·비용↓. 쓰기 경합 없음(웹은 자체 `web_review_cache`에만 씀) |
| ④ AWS Budget 알림 | 운영 중 | 월 예산 50/80/100% 알림 + 킬 스위치(`LLM_COMMENTARY_ENABLED=false`). 링크 유출 시 최후 방어 |
| ⑤ per-identity 일일 쿼터 | **Phase 6 백로그** | 확산 시 도입. ③의 사용량 카운터(아래 결정 4)를 그대로 재사용하면 재구현 불필요 |

**바리케이드(생년월일 등) 미채택 근거**: 추측·무차별 대입에 취약하고 개인정보를 서버가 보관하는 부담. 지인 대상엔 초대코드가 우월하다.

**방어 강화 트리거**: 월 예산 50% 도달, 일 신규장소 요청 급증(예: 100건/일 초과), 초대코드 유출 정황 → 쿼터 도입 / 모델 티어 다운(`ANTHROPIC_MODEL` 상수) / 신규 초대 동결 / 킬 스위치 순으로 검토.

**비용 노브(상시 가용)**: `REVIEW_FETCH_LIMIT` 축소, 모델 티어 다운. 프롬프트 캐싱은 무의미(시스템 프롬프트가 캐시 최소 토큰 미만 + 리뷰는 장소마다 상이) — **DynamoDB 장소 캐시가 유일한 실질 절감선**.

## 결정 4 — 관리자 사용량 통계 (Phase 6 쿼터의 골격)

- 요청당 `web_usage`(PK `identity`) 항목에 누적 합계(`total_count`·`llm_call_count`·`last_used_at`)와 **일별 최상위 카운터**(`req#YYYY-MM-DD`·`llm#YYYY-MM-DD`)를 단일 `update_item`의 `ADD`로 누적한다. DynamoDB `ADD`는 없는 숫자 속성을 0으로 자동 초기화 → 스키마리스라 테이블·템플릿 변경 불필요.
- `llm_call_count`(캐시 미스 = 실제 과금)가 비용 프록시. Phase 6 일일 쿼터는 일별 카운터에 임계값만 얹으면 된다 — 카운터 재구현 불필요.
- `/admin/stats`는 `WEB_ADMIN_TOKEN`으로 보호. identity는 초대코드 표시이름(PII 아님). 리뷰 본문·원문 식별자는 통계에 남기지 않는다(CLAUDE.md 제약 7).
- 주의: 일별 카운터 도입(2026-07-07) 이전 사용량에는 일자 데이터가 없어 구간 필터 시 0으로 보인다(누적 합계에는 반영됨).

## 결정 5 — 프론트: 정적 Next.js PWA + Vercel

**채택**: Next.js 15(App Router)·Tailwind·shadcn/ui 정적 export를 Vercel(Hobby)에 배포. `review_analyst`의 JSON 계약(PRD §4)을 카드 UI로 직접 렌더(MarkdownV2 우회).

**근거**: 설치 불필요·링크 하나로 접근(지인 접근성 최고), 사용자 기본 스택과 일치해 공수 최소, AWS 인프라와 물리 분리(격리 보너스). 정적 파일이라 추후 AWS 이전도 저비용.

## 결정 6 — 장소 텍스트 검색: LLM은 검색어 변환만, 후보는 네이버 결과만 (2026-07-17)

URL을 모르는 사용자를 위한 진입 경로. 자연어 프롬프트 → 네이버 장소 검색 → 후보 리스트 → 클릭 시 `place_id` 직행 분석. 배포된 계약·인프라는 `PRD.md` §10-1·`ARCHITECTURE.md`가 원천이며, 여기서는 설계 선택의 근거만 담는다.

**① LLM 역할을 검색어 정규화로 제한 (환각 0)**: LLM은 자연어를 네이버가 소화하는 "지역+업종/상호" 검색어로 축약만 하고, 사용자에게 보여줄 후보 리스트는 **네이버 instant-search 결과만** 사용한다. LLM이 장소를 직접 지어내지 않으므로 존재하지 않는 식당 추천(환각)이 구조적으로 불가능하다.
- **거부한 대안 — 자연어를 네이버에 그대로 검색**: instant-search는 autocomplete 성격이라 "강남 데이트 양식집" 같은 상황어 섞인 다중개념 질의는 빈 결과를 준다(실측). "강남 양식" 수준으로 정규화해야 결과가 나온다 — 근거 `experiments/findings.md` §6-3. 따라서 LLM 정규화 전처리가 필요하다.

**② `/search`는 동기 응답 (잡+폴링 미사용)**: 검색 1회 = LLM 정규화 1회(Haiku, ~수 초) + 네이버 검색 1요청으로, 캐시 미스 분석 파이프라인(~30초, 결정 2)과 달리 API Gateway 30초 안에 끝난다. 잡 생성·폴링 오버헤드가 불필요하여 `POST /search`는 WebApiFunction이 동기 처리한다. 대신 정규화 LLM 5초 여유를 위해 함수 Timeout을 10→20초로 상향(`ARCHITECTURE.md` 웹 인프라).

**③ 검색 전용 킬 스위치 (`SearchLlmEnabled`)**: 결정 3-④의 분석 킬 스위치(`LLM_COMMENTARY_ENABLED`)와 **독립**. off로 두면 LLM 정규화만 끄고 원문 키워드로 네이버 검색은 그대로 동작한다 — 검색 기능 자체를 죽이지 않고 LLM 비용만 차단하는 세밀한 노브. LLM 정규화는 개선 수단이지 필수 경로가 아니므로(전 실패 시 원문 폴백) off여도 사용자 경험 저하가 제한적이다.

**④ 검색 카운터로 비용 관측 (`search_count`)**: 결정 4의 사용량 통계에 `search_count`(누적)·`search#YYYY-MM-DD`(일별)를 스키마리스 ADD로 더한다. 검색 유입이 LLM 정규화 호출량과 직결되므로, 분석 카운터(`llm_call_count`)와 분리 관측해 검색 경로의 비용·오남용을 조기 포착한다. Phase 6 쿼터 도입 시 검색 임계값도 이 카운터에 얹으면 된다.

## 결정 7 — 응답 지연 최적화 묶음 (2026-07-17)

웹은 Telegram과 달리 사용자가 화면 앞에서 결과를 기다리므로 체감 지연이 곧 이탈로 이어진다. 아래 다섯 결정으로 캐시 히트·미스·검색·콜드스타트 전 구간의 지연을 줄였다. 배포된 코드·인프라 사실은 `ARCHITECTURE.md`가 원천이며, 여기서는 선택의 근거만 담는다.

**① 캐시 히트 API 직결 (계약 무변경 202 유지)**: `/analyze`가 `place_id` 경로(force_refresh 아님)일 때 WebApiFunction이 공유 캐시(`lookup_cached_summary`, web→prod read-through)를 직접 조회한다. 히트면 워커를 invoke하지 않고 done 상태 잡을 즉시 생성한다.
- **거부한 대안 — 히트 시 결과를 200으로 동기 반환**: 프론트가 "202면 폴링 / 200이면 즉시 렌더"로 응답 형태를 분기해야 해 계약이 복잡해진다. 대신 히트에도 종전과 동일하게 `job_id`를 202로 돌려주되 잡을 이미 done 상태로 만들어, 프론트의 다음 폴링 1회가 곧바로 완료 결과를 받게 했다. 프론트 코드·API 계약 무변경, 워커 왕복(비동기 invoke + 콜드 가능성)만 제거 — 캐시 히트 응답이 워커 경유 없이 끝난다.
- `naver_url` 경로는 place_id 해석이 선행되어야 하므로 종전대로 워커행. 직결은 검색 후보 클릭 등 place_id를 이미 아는 경로의 최적화다.

**② 프론트 폴링 백오프**: `useAnalysis`가 첫 조회 250ms → 750ms → 이후 1500ms 정속으로 폴링한다(총 예산 59.5초). 종전의 고정 1.5초 선대기를 제거했다. 캐시 히트(①)는 백엔드가 즉시 완료 상태라 첫 250ms 폴링에서 바로 결과가 잡히고, 미스는 어차피 십수 초가 걸려 후반 정속 구간이 부담 없다 — 짧은 초기 간격이 히트 체감만 개선하고 미스 부하는 늘리지 않는다.

**③ 리뷰 분석 모델 Haiku 4.5 전환**: `ANTHROPIC_MODEL`을 `claude-haiku-4-5`로 내렸다. 동일 리뷰 48건 실측 3회 평균 **Sonnet 4.5 16.5초 → Haiku 9.8초(42% 단축), 요약 품질 동등**. 리뷰 요약은 추론 난도가 낮아 상위 티어의 이점이 지연 대비 미미하다는 판단.
- **A/B 중 발견한 프롬프트 결함**: 모델 교체 검증 과정에서, 기존 프롬프트가 `menus`에 장소 통계 라벨·수치를 그대로 복사해 넣던 결함을 발견했다. 실제 메뉴명과 리뷰 언급 횟수만 쓰도록 프롬프트를 명시화하고, 코드에서 `mentions` 내림차순 방어 정렬을 추가했다.
- anthropic 클라이언트는 timeout 60초·재시도 0으로 유계화(지연 폭주 차단).
- **Sonnet 5 제외**: Sonnet 5는 기본 thinking 동작이라 현행 비스트리밍 호출과 비호환(검증 실패)이라 제외했다.
- 주의: `ANTHROPIC_MODEL`은 Telegram과 공유하는 상수라, Telegram 스택은 재배포 전까지 종전 모델로 동작한다.

**④ 워밍 핑 (대안 비교 후 채택)**: EventBridge Schedule(`rate(5 minutes)`, `{"warmup": true}`)로 두 Lambda를 상시 워밍하고, 핸들러 최상단에서 즉시 반환한다. 콜드스타트 실측 14.6초(/search 기준) 대응.
- **거부한 대안 — 메모리 상향**: CPU 비례 배정이라 콜드스타트의 import 구간은 빨라지지만, 워밍 핑으로 콜드스타트 자체가 드물어지면 한계효과가 작다(비용은 호출당 GB-초 비례로 미미하나 얻는 것도 미미).
- **거부한 대안 — Provisioned Concurrency**: 콜드스타트를 확실히 없애지만 개인용 저트래픽에는 상시 예약 요금이 과하다. 5분 핑은 비용이 사실상 0(짧은 즉시 반환 호출)이면서 상시 워밍 효과를 얻는다.

**⑤ 단계별 latency 계측**: 워커 파이프라인(cache/collect/llm/save)·검색(normalize/search/usage) 소요를 INFO 1줄로 로깅한다(PII 없음). 병목을 실측으로 짚기 위한 관측 장치. 이 과정에서 루트 로거 레벨이 명시되지 않아 Lambda 기본 WARNING에서 INFO 로그가 나오지 않던 문제를 함께 고쳤다(`config.py`에서 INFO 명시).
