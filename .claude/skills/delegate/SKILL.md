# Worker 위임 표준 (Advisor 전용)

구현 노동은 Worker에게, 설계·검증·커밋은 Advisor가. 이 절차를 벗어난 Advisor 직접 구현은 **한두 줄 마무리에만** 허용된다.

## 1. Worker 선택

| 작업 | Worker |
|---|---|
| 코드 작성·수정·테스트 (백엔드·프론트·인프라 템플릿) | `worker-dev` |
| 네이버 응답 실측·엔드포인트 변경 감지 | `worker-scraper` (위임 전 `experiments/findings.md` 확인 — 이미 확정된 사실이면 위임 불필요) |

## 2. 브리프 작성 (아래 5개 절 필수)

```
## 배경
왜 이 작업을 하는지 1~3문장. 관련 기존 코드·문서 경로.

## 범위 (이 파일들만)
수정 대상 파일을 명시적으로 나열. "범위 밖 리팩터링 금지" 명시.

## 요구사항
동작 단위로 번호 매겨 서술. 모호한 형용사 대신 입력→출력 예시.
설계가 이미 결정된 부분은 결정 사항으로 못박는다 (Worker가 재설계하지 않도록).

## 준수 제약
CLAUDE.md "하드 제약" 중 이 작업에 직접 걸리는 항목 번호를 지목.
(예: "제약 1·2·8 적용. 스크래핑 경로 건드리면 11~14도.")

## 완료 기준
통과해야 할 테스트·빌드 명령을 정확히. (예: pytest tests/test_web_store.py 신규 케이스 포함 그린,
web-frontend 변경 시 npm run build && npm run lint 그린)

## 보고 형식
변경 파일 목록 / 핵심 결정 / 실행한 명령과 결과 전문 / 브리프와 달리 한 부분과 이유
```

## 3. 병렬 위임 판단

- 파일이 겹치지 않는 독립 작업(예: 백엔드 `src/` vs 프론트 `web-frontend/`)은 **2개 병렬** 위임
- 프론트-백엔드 간 **계약(API 필드명·형태)은 브리프에 양쪽 동일하게 명시**하고, 완료 후 Advisor가 계약 정합을 직접 대조한다
- 같은 파일을 만지는 작업은 병렬 금지 — 순차 위임

## 4. 완료 보고 수신 후 (검증·승인)

1. **보고를 믿지 말 것** — `git diff`를 직접 읽는다 (범위 준수·요구사항 반영 확인)
2. `/verify` 스킬 실행 (테스트·빌드·제약 위반 스캔)
3. **실패 시**: 직접 고치지 말고 실패 내용·기대 동작을 담은 **수정 브리프로 재위임** (같은 Worker에 SendMessage로 이어가면 컨텍스트 유지)
4. **통과 시**: 작은 단위로 커밋 (한국어 메시지), 사용자에게 결과 보고

## 5. Worker가 설계 판단을 반환했을 때

Worker는 브리프에 없는 설계 판단을 임의로 하지 않고 선택지와 함께 반환한다. Advisor가 결정해 브리프를 보강한 뒤 재위임한다 (Worker에게 결정을 떠넘기지 않는다).
